# Week 2 Progress – Event Knowledge Graph Pipeline

## Overview

This week I focused on building the **initial data pipeline for the Event Knowledge Graph** used in the Digital Twin platform.
The goal was to enable **real-time event ingestion from MQTT and automatic storage in a Neo4j graph database**.

The implemented pipeline connects event streams from the LEGO factory simulation to a structured **Event Knowledge Graph**, following the schema discussed in the research group (Event–Activity–Entity model).

Current pipeline:

```
MQTT Event Stream
        ↓
record_events.py (MQTT consumer)
        ↓
write_event_to_graph()
        ↓
Neo4j Event Knowledge Graph
```

This allows events generated by the system to be automatically transformed into graph structures suitable for **process mining and digital twin analysis**.

---

## What I Implemented

### 1. Event Knowledge Graph Schema

Implemented the graph schema in **Neo4j** including the following node types:

```
Event
Activity
Entity
EntityType
Station
```

Relationships:

```
(Event)-[:OF_ACTIVITY]->(Activity)
(Event)-[:ACTS_ON]->(Entity)
(Entity)-[:OF_TYPE]->(EntityType)
(Event)-[:OCCURRED_AT]->(Station)
(Event)-[:DF]->(Event)
```

The **DF (Directly Follows)** relationship represents the temporal order of events for the same part.

---

### 2. MQTT Event Ingestion

Implemented a **MQTT consumer** that subscribes to the event stream:

```
component_event/+/all
```

Key features:

- Automatic message subscription
- JSON payload parsing
- Real-time event processing
- Integration with Neo4j writer

Main file: `record_events.py`

When a message is received:

```
MQTT message
     ↓
deserialize JSON event
     ↓
append to event log
     ↓
write_event_to_graph(event)
```

---

### 3. Neo4j Graph Writer

Implemented a **graph writer module** that converts event data into graph structures.

File: `neo4j_writer.py`

Main responsibilities:

- Create Event nodes
- Create related entities (Station, Entity, Activity)
- Maintain DF relationships between consecutive events of the same part

Example event:

```json
{
  "time": 1,
  "station_id": "S1",
  "part_id": "P1",
  "part_type": "part",
  "activity": "START"
}
```

Automatically generates graph nodes and relationships.

---

### 4. Event Simulation

Implemented a **test event producer** to simulate factory events.

File: `send_test_events.py`

This script publishes events to the MQTT broker:

```
component_event/test/all
```

Example sequence:

```
START → PROCESS → END
```

Each event is sent with a small delay to simulate a real event stream.

---

### 5. Event Log Backup

All received events are stored in memory and saved to a CSV file when the program stops.

Example output:

```
logs/event_log_240305_142530.csv
```

This provides a **backup event log for offline analysis or process mining experiments**.

## Current Graph Structure

Example Event Knowledge Graph generated in Neo4j:

- Event nodes connected through **DF relationships**
- Each event linked to:
  - Activity
  - Station
  - Entity
  - EntityType

This structure follows the **Event Knowledge Graph model used for process mining and digital twins**.
